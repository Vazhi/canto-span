# Workflow

## 1. Select a bounded question

Choose an exact construction family or internal representation problem. Rank candidates by pattern-specific authority, source independence, natural evidence, negative boundaries, competing analyses, native-conflict status, learner value, architectural isolation, and fresh held-out feasibility. Do not begin from a desired grammar rule.

## 2. Build authority-origin provenance

Extract exact source propositions and locators. Classify each relationship as support, restriction, contradiction, competing analysis, attestation, or background. Record limitations and forbidden inferences. A source gap is an acceptable result and must not be disguised as support.

## 3. Define the work package

State:

- exact target and non-targets;
- intended semantic change;
- permitted and forbidden behavior changes;
- positive, negative, ambiguous, preservation, and render cases;
- source propositions and competing analyses;
- acceptance, revision, rejection, quarantine, retirement, and rollback criteria.

## 4. Freeze evaluation before implementation

Create and lock the visible packet, development baseline, render targets, public held-out commitments and hashes, and a sealed evaluator archive outside the recovery tree. Do not inspect hidden surfaces or salts.

## 5. Implement the smallest evidence-faithful change

Prefer a narrow internal subtype when a broad legacy label overstates the evidence. Preserve compatibility only when it does not broaden the claim. Keep grammar, lexicon, Jyutping, learner-role, and internal-representation changes separately attributable.

## 6. Run headless validation

Run the applicable gates defined in [`VALIDATION-AND-ACCEPTANCE.md`](VALIDATION-AND-ACCEPTANCE.md). Do not edit snapshots or expected outputs merely to force a pass. Failures must be explained as intended changes, regressions, evidence problems, or test defects.

## 7. Render and adjudicate

Install the exact three-file runtime candidate and inspect every focused row and complete tree. Verify spans, roles, Jyutping, hover text, context state, root coverage, legitimacy, ambiguity, and boundaries. Machine summaries alone cannot accept learner-visible behavior.

## 8. Freeze the exact candidate

After render approval, hash runtime, packet, diagnostics, project state, and evaluator custody. No parser change is allowed after this freeze. Any behavioral edit creates a new candidate and requires fresh prospective held-out commitments.

## 9. Open held-out custody once

Verify commitments, evaluate the frozen candidate, record results, and consume every revealed case. Revealed cases become ordinary regression evidence and can never again serve as prospective held out.

## 10. Adjudicate and package

Choose only an allowed disposition: accept, revise, reject, quarantine, or retire. Update all state-owning documents atomically, then build the separated packages defined in [`PACKAGING-AND-RECOVERY.md`](PACKAGING-AND-RECOVERY.md).

## Batch and checkpoint policy

Work batches are sized by coherent research or implementation conclusions, not rigid source, example, or time caps. Create a downloadable recovery checkpoint after each substantial batch and before implementation, evaluator opening, major branch expansion, or context-risk. A checkpoint must include a current ZIP and a concise machine-readable or Markdown state ledger.
