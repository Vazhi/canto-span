# Validation and acceptance

## Gate families

| Gate | Purpose |
|---|---|
| Packet lock | Confirms scope, visible cases, render cases, commitments, and custody were fixed before implementation. |
| Focused evaluation | Tests the exact work-package contract. |
| Preservation | Protects accepted neighboring behavior and compatibility contracts. |
| Aggregate regression | Detects changes to inherited accepted snapshots. |
| Grammar legitimacy | Verifies honest claim classes, statuses, aliases, and non-licensing internal labels. |
| Semantic acceptance | Prevents malformed, unresolved, or incomplete output from being reported as clean. |
| Corpus evaluation | Exposes coverage and false analyses without treating corpus retrieval as proof. |
| Source accounting | Accounts for every active language and internal label. |
| Claim provenance | Verifies proposition-bearing source edges, locators, limitations, and authority origin. |
| Native-conflict burden | Enforces the higher burden for source–naturalness conflict. |
| Native naturalness | Keeps surface judgments separate from parser-analysis validation. |
| Topology and preservation | Protects accepted construction structure while permitting explicitly bounded lexical or internal work. |
| Render review | Adjudicates learner-visible spans, roles, Jyutping, glosses, context, and full diagnostic trees. |
| Prospective held-out | Tests the exact frozen candidate once on unseen committed cases. |
| Documentation and packaging | Verifies current-state consistency, valid links/JSON, archive classification, and package separation. |

Not every internal cleanup needs natural-attestation promotion evidence, but it must prove non-licensing and semantic neutrality. Every language-claim promotion must satisfy the full applicable evidence burden in `DOCTRINE.md`, including all conflict-escalation requirements when a native conflict exists.

## Productive promotion requirements

A language family may become `supported_productive` only when all applicable requirements pass:

1. narrow source-originated claim;
2. independent scope-matched positive evidence;
3. the default natural-attestation minimum or a documented stronger alternative design;
4. controlled variation and explicit negative boundaries;
5. competing-analysis review;
6. no unresolved native conflict on the exact family;
7. frozen visible and render packets;
8. learner-facing render approval;
9. one-time prospective held-out success;
10. preservation, regression, semantic, corpus, provenance, conflict, documentation, and packaging gates;
11. exact candidate freeze before held-out opening.

## Dispositions

- **ACCEPT** — all applicable gates pass; accepted scope and limitations are explicit.
- **REVISE** — useful work exists but one or more requirements fail; opened held-out cases remain consumed.
- **REJECT** — evidence, accepted behavior, or learner-facing accuracy fails materially.
- **QUARANTINE** — retain the item for visibility or research while blocking promotion or parser licensing.
- **RETIRE** — remove an indefensible or redundant language claim or internal representation while preserving history.

A failure cannot be converted into a pass by widening the claim, weakening the boundary, rewriting the hidden packet, or updating snapshots without an independently justified behavior change.

Current pass counts and accepted scope are owned by [`PROJECT-STATE.md`](PROJECT-STATE.md), not duplicated here.
